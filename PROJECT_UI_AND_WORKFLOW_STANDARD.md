# Personal Hub 设计规范

更新时间：2026-06-29

本文档只定义项目的界面设计规范，用于后续所有网页端和移动端 UI 设计、样式调整、组件优化和页面改版。本文档不描述具体业务功能，不规划功能模块，不规定某个页面必须包含什么业务内容。

## 1. 设计目标

项目界面应呈现为一个长期使用的数据工作台，而不是营销官网或展示型页面。

核心目标：

- 清晰：用户能快速看懂当前页面的结构、重点和状态。
- 高效：常用操作容易找到，录入、筛选、查看都不绕路。
- 克制：减少装饰性元素，把视觉注意力留给内容和数据。
- 统一：不同页面使用同一套颜色、字号、间距、组件和状态。
- 稳定：桌面端和移动端都不破版、不溢出、不跳动。
- 可维护：新增页面和组件时优先复用现有规范。

设计关键词：

- 现代
- 克制
- 清晰
- 专业
- 数据优先
- 高密度但不拥挤
- 桌面端与移动端一致体验

避免方向：

- 不做营销官网式大横幅。
- 不做大面积装饰性渐变。
- 不做过度圆角和厚重阴影。
- 不做每个页面一套独立风格。
- 不做只适合桌面端、移动端勉强缩放的界面。

## 2. 设计原则

### 2.1 一致性

同一种控件、同一种状态、同一种布局结构，在全项目中应保持一致。

要求：

- 同类按钮高度一致。
- 同类输入框高度一致。
- 同类卡片内边距一致。
- 同类标签颜色和形状一致。
- 页面标题、工具栏、内容区的结构一致。

### 2.2 层级清晰

页面视觉层级应帮助用户判断优先级。

要求：

- 页面标题明显，但不夸张。
- 关键数据比说明文字更突出。
- 主操作比次要操作更突出。
- 危险操作与普通操作明显区分。
- 辅助信息弱化，但仍可读。

### 2.3 信息密度适中

项目属于工具型界面，应支持较高信息密度，但不能拥挤。

要求：

- 卡片之间留出稳定间距。
- 表单项之间保持清晰分组。
- 长列表和表格适合快速扫描。
- 不使用大面积空白造成信息稀疏。
- 不把过多内容堆进同一个卡片。

### 2.4 响应式优先

设计时同时考虑桌面端、平板和手机。

要求：

- 桌面端可以多列展示。
- 平板端减少列数。
- 手机端优先单列布局。
- 移动端不出现页面级横向滚动。
- 固定格式组件要有稳定尺寸。

## 3. 颜色规范

当前项目以深色工作台风格为主，同时保留浅色主题基础。

### 3.1 主色变量

推荐使用以下颜色变量作为全局基础：

```css
:root {
  --bg: #0f1014;
  --surface: #171922;
  --surface-2: #20232d;
  --surface-3: #272b36;
  --card: #1b1d25;
  --text: #f4f5f7;
  --muted: #9ca3af;
  --weak: #6b7280;
  --line: #2c2f3a;
  --strong-line: #3a3f4f;
  --accent: #ff1b2d;
  --accent-2: #5b8cff;
  --success: #30d158;
  --warning: #ffb020;
  --danger: #ff4d5e;
}
```

### 3.2 使用规则

- 背景色用于页面底层，不直接承载大量内容。
- 表面色用于导航、工具栏、面板。
- 卡片色用于主要内容容器。
- 边框色用于区分层级，避免厚重分割线。
- 强调色只用于当前状态、主操作或关键提示。
- 成功、警告、危险色必须语义清晰，不混用。
- 不只依靠颜色表达状态，应配合文字、图标或形状。

### 3.3 禁止项

- 禁止大面积高饱和纯色背景。
- 禁止大面积紫蓝渐变。
- 禁止每个页面随意新增主色。
- 禁止普通提示滥用红色。
- 禁止同一状态在不同页面使用不同颜色。

## 4. 字体规范

### 4.1 字号层级

| 场景 | 字号 | 字重 | 用途 |
| --- | --- | --- | --- |
| 页面标题 | 22px - 26px | 700 | 当前页面或区域名称 |
| 模块标题 | 18px - 20px | 700 | 主要内容分区标题 |
| 卡片标题 | 15px - 17px | 600 / 700 | 卡片、列表项标题 |
| 正文 | 14px - 16px | 400 / 500 | 表单、说明、明细 |
| 辅助文字 | 12px - 13px | 400 | 日期、备注、状态说明 |
| 数据数字 | 20px - 28px | 700 | 金额、数量、统计值 |
| 按钮文字 | 13px - 14px | 600 | 操作按钮 |

### 4.2 排版规则

- 不使用负字距。
- 不用超大标题挤压内容空间。
- 不用随视口宽度变化的字体算法。
- 中文正文保持清晰行高，避免过密。
- 数字信息可以突出，但不能造成卡片高度跳动。
- 移动端字体不应小于 12px。

## 5. 间距规范

统一使用 4px / 8px 间距体系。

推荐间距：

```text
4px
8px
12px
16px
20px
24px
32px
40px
48px
```

使用规则：

- 页面左右边距：桌面端根据容器宽度控制，移动端推荐 16px。
- 卡片内边距：推荐 16px / 20px / 24px。
- 卡片之间：推荐 16px / 20px / 24px。
- 页面大区块之间：推荐 24px / 32px。
- 紧密列表项之间：推荐 8px / 12px。
- 工具栏内控件之间：推荐 8px / 12px。

## 6. 圆角、边框和阴影

### 6.1 圆角

推荐：

```text
小按钮：8px
输入框：10px
普通卡片：12px
大型容器：16px
弹窗：16px
图标按钮：10px - 12px
```

规则：

- 圆角应统一，不要同页面出现过多圆角尺寸。
- 工具型界面避免过度圆润。
- 小控件圆角小，大容器圆角稍大。

### 6.2 边框

规则：

- 普通分隔使用 1px 低对比边框。
- 重要状态可以使用更明显边框。
- 不使用厚重边框制造层级。
- 边框颜色优先使用变量。

推荐：

```css
border: 1px solid var(--line);
```

### 6.3 阴影

规则：

- 普通卡片不依赖厚重阴影。
- 弹窗、浮层、当前突出面板可以使用阴影。
- 阴影要轻，不制造漂浮感过强的页面。

## 7. 布局规范

### 7.1 页面结构

推荐页面结构：

```text
导航区
页面标题区
工具栏 / 筛选区
主要内容区
辅助内容区
```

规则：

- 页面标题区位置稳定。
- 主操作放在标题区或工具栏右侧。
- 搜索、筛选、排序放在同一工具区。
- 内容区使用统一栅格或统一列表结构。
- 页面最大宽度要受控，避免超宽屏内容被拉散。

### 7.2 栅格

桌面端：

- 可使用 2 到 4 列卡片布局。
- 数据密集区域可以使用表格或紧凑列表。
- 主要内容和辅助内容应有明确比例。

平板端：

- 优先减少列数。
- 避免三列以上拥挤布局。

移动端：

- 优先单列。
- 重要操作靠近内容。
- 不使用需要横向理解的复杂布局。

## 8. 页面优先级与标准结构

### 8.1 核心页面职责

后续页面改版先判断页面职责，再决定首屏展示内容和操作优先级。

| 页面 | 主要职责 | 首屏优先展示 | 低频内容归属 |
| --- | --- | --- | --- |
| 总览 | 每日指挥中心 | 关键指标、今日行动、风险提醒 | 趋势、最近访问、历史复盘 |
| 生活收支 | 财务数据中心 | 本月结余、预算节奏、异常流水、复核入口 | 导入配置、月报历史、长期预测 |
| 订阅 | 续费与省钱决策中心 | 临期订阅、月均/年化成本、复盘队列 | 通知配置、分布细节、历史记录 |
| 人情往来 | 关系与待回礼提醒中心 | 关系差额、待回礼、最近往来、重点人物 | 事件模板、导入校验、长期统计 |
| 事项 | 日常执行中心 | 今日、逾期、进行中、完成进度 | 历史复盘、批量整理 |
| 设置 | 低频管理与数据安全中心 | 同步状态、备份状态、账号状态 | 管理员工具、危险操作、导入导出 |

### 8.2 页面标准结构

推荐所有核心页面尽量遵循同一信息节奏：

```text
页面标题区
关键指标区
行动 / 风险提醒区
主要明细区
低频工具区
```

规则：

- 页面首屏只放最能指导下一步行动的信息。
- 关键指标不超过 4 到 6 个，避免指标堆砌。
- 行动和风险提醒要能直接跳转到处理位置。
- 明细区用于浏览和筛选，不承担所有解释性内容。
- 导入、配置、清空、管理等低频工具应放在侧栏、折叠区、设置页或低频工具区。
- 移动端优先保留关键指标和行动入口，复杂图表与低频工具下移。

### 8.3 统一页面组件

核心页面优先复用以下组件语义：

- Section header：用于区块标题、说明和右侧动作。
- KPI strip：用于横向关键指标，不替代完整图表。
- Action / risk panel：用于今日行动、待复核、临期和异常提醒。
- Record list：用于事项、账单、订阅、人情等明细。
- Tools panel：用于低频操作、导入导出、设置和管理动作。

禁止项：

- 不为每个页面重新设计一套卡片、按钮、标签和筛选器。
- 不把所有功能都放在首屏。
- 不让低频管理功能抢占主要行动区域。

## 9. 组件规范

### 9.1 按钮

按钮类型：

- 主按钮：用于主要确认动作。
- 次按钮：用于普通操作。
- 危险按钮：用于高风险操作。
- 图标按钮：用于小范围快捷操作。
- 文本按钮：用于弱操作或链接式操作。

规则：

- 同一区域最多一个主按钮。
- 同一按钮组高度一致。
- 按钮文字应短而明确。
- 图标按钮需要明确含义。
- 危险按钮必须与普通按钮明显区分。
- 禁用按钮必须明显不可点击。

推荐高度：

```text
小按钮：32px
常规按钮：36px - 40px
移动端主要按钮：40px - 44px
```

### 9.2 输入控件

包括输入框、搜索框、下拉框、日期选择等。

规则：

- 高度保持一致，推荐 40px - 44px。
- label、输入框、帮助文字、错误提示形成固定结构。
- 聚焦状态清晰。
- 错误状态靠近对应输入项。
- 占位文字不能代替 label。
- 移动端输入区域不能过窄。

### 9.3 卡片

规则：

- 卡片用于承载独立信息。
- 避免卡片套卡片。
- 同类卡片结构一致。
- 卡片标题、主要内容、辅助信息层级清楚。
- 卡片操作按钮靠近对应内容。
- 移动端卡片紧凑但必须可读。

### 9.4 标签和徽章

规则：

- 标签文字要短。
- 同类标签颜色一致。
- 状态标签和分类标签不能混淆。
- 标签不应抢过主要内容的视觉权重。

### 9.5 表格和列表

桌面端：

- 表格适合密集信息。
- 表头固定语义清楚。
- 列宽稳定。
- 金额、日期、状态等内容易扫描。

移动端：

- 优先改为卡片式列表。
- 每条记录突出主信息。
- 次要信息弱化或折叠。
- 操作按钮靠近对应记录。

### 9.6 弹窗

规则：

- 弹窗用于编辑、详情、确认等短流程。
- 标题必须明确。
- 主操作和取消操作位置稳定。
- 危险确认流程不能轻易误触。
- 移动端弹窗不超过屏幕宽度。
- 弹窗内容过长时内部滚动，不撑破页面。

## 10. 交互状态规范

所有可交互控件都应考虑以下状态：

- 默认
- 悬停
- 按下
- 聚焦
- 禁用
- 加载中
- 成功
- 失败
- 警告

规则：

- 用户操作后必须有反馈。
- 保存、提交、删除、导入等操作需要明确结果。
- 加载中状态不能让用户误以为页面卡死。
- 失败提示要说明原因和下一步。
- 危险操作需要二次确认。
- 可以撤销的操作优先提供撤销。

## 11. 空状态和边界状态

每个页面和组件都应考虑：

- 无数据
- 加载中
- 加载失败
- 权限不足
- 网络失败
- 长文本
- 超大数字
- 异常值
- 图片或图标缺失
- 移动端窄屏

空状态规则：

- 说明当前没有内容。
- 提供下一步操作。
- 不使用大段解释文字。
- 不让空页面显得像错误页面。

## 12. 移动端规范

### 12.1 尺寸

移动端验收尺寸：

```text
320px
375px
390px
414px
768px
```

规则：

- 页面不出现横向滚动。
- 左右安全边距推荐 16px。
- 点击区域尽量不小于 44px。
- 顶部和底部固定区域不能遮挡主要内容。
- 弹窗、菜单、筛选区必须适配窄屏。

### 12.2 布局

- 优先单列布局。
- 多列卡片在窄屏下改为单列。
- 表格在窄屏下改为卡片列表。
- 筛选项不要堆得过高。
- 重要操作放在容易触达的位置。

### 12.3 文本

- 长标题允许换行。
- 长数字不能撑破容器。
- 按钮文字不能溢出按钮。
- 标签文字过长时截断或换行。

## 13. 可访问性规范

基础要求：

- 可点击元素有 focus 状态。
- 图标按钮需要 title 或 aria-label。
- 表单控件需要 label。
- 状态不能只靠颜色表达。
- 文本和背景保持足够对比度。
- 键盘用户应能完成主要操作。
- 错误提示应清楚可理解。

## 14. 文案规范

规则：

- 按钮使用动词。
- 提示文案短而明确。
- 错误文案说明原因和下一步。
- 空状态文案说明当前状态和可做操作。
- 同一概念使用同一名称。
- 中文界面避免无意义混入英文。

## 15. UI 验收清单

每次 UI 改动完成后检查：

- 桌面端布局是否稳定。
- 移动端是否无页面级横向滚动。
- 长文本是否不会撑破容器。
- 按钮高度是否统一。
- 输入框高度是否统一。
- 卡片间距是否统一。
- 主操作是否清楚。
- 危险操作是否明显。
- 空状态是否存在。
- 加载状态是否存在。
- 错误状态是否清楚。
- hover、active、focus、disabled 是否完整。
- 深色主题下文字对比是否足够。
- 移动端点击区域是否足够。

## 15. 后续执行约定

从本文档建立后，后续所有设计和 UI 实现默认遵守：

- 先复用现有设计变量。
- 先复用现有组件样式。
- 新增样式必须符合颜色、字号、间距、圆角规范。
- 改桌面端时同步考虑移动端。
- 改组件时同步考虑交互状态。
- 改页面时同步检查空状态和边界状态。
- 如果需求与本文档冲突，先更新设计规范，再执行 UI 改动。
