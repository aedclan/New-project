# Personal Hub 优化路线图

这份路线图用于把当前项目从“功能完整”推进到“体验统一、数据可信、移动端好用、工程可维护”。建议按阶段执行，不要一次性大改。

## 总目标

- 把 Dashboard 做成每日指挥中心。
- 把账单模块做成可信的数据中心。
- 把订阅模块做成省钱决策中心。
- 把人情模块做成关系提醒中心。
- 把移动端做成快速查看和快速录入中心。
- 在不重写项目的前提下，逐步统一 UI、优化数据质量、补足测试。

## 执行原则

- 先统一体验，再增加新功能。
- 先优化高频页面，再处理低频管理功能。
- 先做可感知改进，再做底层重构。
- 每次只改一个模块，改完运行 `npm run check`。
- 不一次性重写 `renderer.js` 和 `store.js`，优先渐进拆分。

## 阶段 1：产品主线和信息架构

目标：明确每个页面的职责，减少功能堆叠。

- [ ] 确定 3 个核心使用场景：今天要处理什么、钱花到哪里了、哪些事项需要提醒。
- [ ] 梳理所有页面：总览、事项、生活收支、订阅、人情、收藏、笔记、项目集、设置。
- [ ] 为每个页面标记高频功能和低频功能。
- [ ] 高频功能放主区域，低频功能收进设置、抽屉、折叠面板或更多菜单。
- [ ] 明确 Dashboard、账单、订阅、人情四个核心模块的首屏信息优先级。

验收标准：

- [ ] 每个页面都有清晰的主任务。
- [ ] 用户打开页面后，3 秒内能知道该页面最重要的信息。
- [ ] 导入、清空、管理、调试类功能不挤占主内容区域。

## 阶段 2：统一 UI 和版式

目标：让项目看起来像同一个产品，而不是多个页面拼接。

- [ ] 统一颜色：主色、成功色、警告色、危险色、弱文本色。
- [ ] 统一间距：页面边距、卡片内边距、列表间距、表单间距。
- [ ] 统一圆角：按钮、卡片、输入框、标签。
- [ ] 统一按钮样式：主按钮、次按钮、危险按钮、图标按钮。
- [ ] 统一标签和状态样式：待处理、进行中、已完成、即将到期、风险、已同步。
- [ ] 减少卡片种类，只保留概览卡、列表卡、详情卡三类基础形态。
- [ ] 统一空状态样式，为每个模块提供新增、导入或查看说明入口。

建议沉淀文档：

- [ ] 在 `DESIGN_LANGUAGE.md` 中补充颜色、间距、按钮和卡片规范。
- [ ] 或新增 `UI_STANDARD.md`，专门记录 UI 使用规则。

验收标准：

- [ ] 不同模块的按钮、卡片、筛选器看起来一致。
- [ ] 页面主次层级清晰，强调色不滥用。
- [ ] 桌面端和移动端都能保持同一套视觉语言。

## 阶段 3：重构 Dashboard

目标：把 Dashboard 做成每日指挥中心。

推荐版式：

1. 顶部：关键指标。
2. 中部：今日行动和风险提醒。
3. 底部：趋势、复盘、最近更新。

任务清单：

- [x] 顶部只保留 3 到 5 个关键指标：本月结余、今日待办、7 天内订阅、待回礼、同步状态。
- [x] 增加“今日需要处理”区域，聚合事项、订阅、人情、账单异常。
- [x] 增加“风险提醒”区域，展示预算超支、未分类账单、即将续费、待回礼。
- [x] 降低纯展示型统计的优先级，把趋势和历史放到底部。
- [x] 移动端 Dashboard 只保留关键指标、今日行动和快速新增。

验收标准：

- [ ] Dashboard 不再只是数据堆叠，而是能指导下一步行动。
- [ ] 首屏信息少而准，用户能快速判断今天该处理什么。
- [ ] 移动端 Dashboard 不拥挤。

## 阶段 4：优化账单模块

目标：把账单模块做成可信的数据中心。

建议拆成 4 个视角：

- 流水：查看所有收入支出记录。
- 预算：查看本月预算、剩余额度、月底预测。
- 固定支出：管理房贷、车贷、保险、分期、订阅等周期性支出。
- 复盘：查看月度总结、异常提醒、AI 分析。

任务清单：

- [ ] 增加账单数据健康提示：未分类、疑似重复、异常金额、缺少付款人、排除分析数量。
- [ ] 增加预算进度：已花金额、剩余额度、按当前速度预计月底支出。
- [ ] 固定支出单独聚合展示，避免只散落在账单列表中。
- [ ] 导入前自动提示备份。
- [ ] 导入后显示审计报告：新增、跳过、重复、失败、未分类数量。
- [ ] 支持批量分类、批量排除分析、批量设置付款人。
- [ ] AI 财务分析展示数据范围和依据，避免结果看起来像凭空生成。

验收标准：

- [ ] 用户能知道当前账单数据是否可靠。
- [ ] 导入后能清楚知道发生了什么。
- [ ] 账单页既能看流水，也能支持预算和复盘。

## 阶段 5：优化订阅模块

目标：把订阅模块做成省钱决策中心。

任务清单：

- [x] 展示月成本、年成本、未来 30 天成本、未来 90 天成本。
- [x] 增加订阅健康状态：高频刚需、低频可砍、长期未使用、即将续费、价格异常。
- [x] 增加复盘流程：最近是否使用、满意度、是否值得继续付费、下一次复盘日期。
- [x] 支持一键生成账单，并在订阅卡片显示最近一次付款。
- [x] 增加续费前操作：续费、暂停、取消、标记已评估、创建提醒事项。
- [x] 邮件通知增加预览，让用户知道会发送什么内容。

验收标准：

- [ ] 用户不只知道什么时候续费，还能判断是否值得续费。
- [ ] 订阅和账单之间有清晰关联。
- [ ] 即将续费和建议取消的项目足够醒目。

## 阶段 6：优化人情往来模块

目标：把人情模块做成关系提醒中心。

任务清单：

- [ ] 以联系人为核心组织页面，而不是只看事件列表。
- [ ] 联系人详情展示送出金额、收到金额、当前差额、最近事件、待回礼状态、备注。
- [ ] 增加待回礼队列：联系人、上次金额、建议金额、最近事件、是否已处理。
- [ ] 增加事件模板：婚礼、生日、乔迁、满月、过年、探望。
- [ ] 支持人情事件和账单互相关联，减少重复录入。
- [ ] 增加关系提醒：重要联系人长期无互动、节日前提醒、生日提醒。

验收标准：

- [ ] 用户能围绕“人”查看完整往来历史。
- [ ] 待回礼事项清晰可见。
- [ ] 人情金额和账单金额可以互相追踪。

## 阶段 7：优化事项模块

目标：让事项模块更适合日常执行，而不是复杂项目管理。

任务清单：

- [ ] 简化事项视图：今日、进行中、逾期、已完成。
- [ ] 当前页面新增事项时，默认继承当前场景。
- [ ] 增加延期原因、完成备注、实际完成时间。
- [ ] 支持从笔记、订阅、人情提醒中创建事项。
- [ ] 完成事项后支持写简短复盘。

验收标准：

- [ ] 用户每天打开事项页能快速知道该做什么。
- [ ] 逾期和延期原因可追踪。
- [ ] 事项能和其他模块联动。

## 阶段 8：优化移动端体验

目标：移动端不照搬桌面端，而是专注查看和快速录入。

任务清单：

- [x] 移动端底部 Tab 只保留：总览、账单、事项、更多。
- [x] 移动端 Dashboard 只显示今日事项、本月结余、即将续费、待回礼、快速新增。
- [x] 新增表单改成分步流程：选择类型、填写核心字段、补充高级字段。
- [x] 账单移动端新增只保留核心字段：金额、类型、分类、日期、备注。
- [x] 移动端列表卡片只显示标题、金额或状态、日期、一个关键标签。
- [x] 详情和高级操作使用底部抽屉。

验收标准：

- [ ] 移动端首屏不拥挤。
- [ ] 新增账单或事项不需要填写过多字段。
- [ ] 高级操作不干扰日常查看。

## 阶段 9：数据安全、同步和备份

目标：让用户放心录入真实数据。

任务清单：

- [ ] 增加同步状态提示：本地模式、已登录同步、正在同步、同步失败、离线待同步。
- [ ] 增加最近备份时间、备份文件数量、一键下载备份。
- [ ] 导入前自动创建备份或明确提示用户备份。
- [ ] 清空数据、恢复备份、批量覆盖、删除账号等危险操作需要二次确认。
- [ ] 危险操作要求输入确认词，例如 `清空` 或 `DELETE`。
- [ ] 管理员操作显示风险提示。

验收标准：

- [ ] 用户随时知道数据是否已同步。
- [ ] 用户知道最近一次备份是什么时候。
- [ ] 危险操作不会被误触发。

## 阶段 10：工程质量和可维护性

目标：减少后续改动引入 bug 的风险。

任务清单：

- [ ] 逐步拆分 `src/views/renderer.js`。
- [ ] 优先拆出 `dashboard-renderer.js`、`bills-renderer.js`、`subscriptions-renderer.js`、`favors-renderer.js`、`settings-renderer.js`。
- [ ] 逐步拆分 `src/core/store.js`。
- [ ] 优先拆出账单、订阅、人情、事项相关逻辑。
- [ ] 建立数据 schema 或数据规范文件，统一默认值、校验、迁移。
- [ ] 增加核心业务测试：账单汇总、预算预测、订阅周期、人情差额、导入去重、数据迁移。
- [ ] 保留并增强 `npm run check`，但不要只依赖它代替测试。
- [ ] 增加 `CHANGELOG.md`，记录每个版本的用户可见变化。

验收标准：

- [ ] 大文件复杂度下降。
- [ ] 核心计算逻辑有测试保护。
- [ ] 版本变化可追踪。

## 推荐 4 周执行节奏

### 第 1 周：统一方向和 Dashboard

- [ ] 梳理页面高频/低频功能。
- [ ] 确定 UI 规范。
- [ ] 画出新版 Dashboard 草图。
- [ ] 调整 Dashboard 顶部关键指标。
- [ ] 调整 Dashboard 今日行动和风险提醒。
- [ ] 测试桌面端和移动端展示。

### 第 2 周：账单模块

- [ ] 重排账单页结构。
- [ ] 增加数据健康提示。
- [ ] 增加预算进度和月底预测。
- [ ] 优化导入审计报告。
- [ ] 增加批量分类或批量排除分析。

### 第 3 周：订阅和人情

- [ ] 订阅增加成本视角。
- [ ] 订阅增加复盘状态和建议取消提示。
- [ ] 人情改成联系人优先。
- [ ] 人情增加待回礼队列。
- [ ] 订阅、人情与账单建立更清晰的关联。

### 第 4 周：移动端、数据安全、测试

- [ ] 简化移动端底部导航。
- [ ] 优化移动端新增表单。
- [ ] 增加同步和备份状态。
- [ ] 加强危险操作确认。
- [ ] 补第一批核心业务测试。

## 暂缓事项

以下事项不建议现在优先做：

- [ ] 立即引入 React、Vue 或大型前端框架。
- [ ] 一次性重写整个前端。
- [ ] 继续大量增加新模块。
- [ ] 在数据质量没有打牢前继续扩展复杂 AI 功能。
- [ ] 让移动端完整复制桌面端功能。

## 每次迭代完成后检查

- [ ] 执行 `npm run check`。
- [ ] 检查桌面端关键页面。
- [ ] 检查移动端关键页面。
- [ ] 检查新增、编辑、删除、导入、同步是否正常。
- [ ] 更新路线图勾选状态。
- [ ] 如有版本发布，更新 `CHANGELOG.md`。
