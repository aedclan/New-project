# Personal Hub 产品升级路线

当前版本：v0.25.0

本文档作为后续功能升级的主路线。后续开发按以下 4 个阶段推进，优先保证稳定、清晰、可长期维护，不再随意发散功能。

## 总体路线

1. V0.22 稳定上线版：让网站先稳定、干净、不会出现明显错误。
2. V0.23 生活收支增强版：把记账升级为家庭消费分析。
3. V0.24 订阅管理成熟版：把订阅升级为续费提醒与成本管理。
4. V0.25 数据安全与真实登录版：把本地原型升级为可长期使用的真实系统。

## V0.22 稳定上线版

### 目标

让网站在线上稳定运行，页面干净统一，不出现明显报错、遮挡、乱码、按钮无反应、导入导出失效等问题。

### 需要完成的内容

#### 1. 线上访问稳定

功能作用：
- 确保 `https://www.aedclan.com` 可以稳定访问。
- 避免页面出现初始化遮罩、白屏、资源加载失败。
- 保证 Docker 容器启动后可通过健康检查确认状态。

实现方法：
- 保留 `/healthz` 健康检查接口。
- 修复静态服务器对异常路径的兼容，例如 `//` 请求。
- 每次部署后执行：
  - `docker compose ps`
  - `docker compose logs -f --tail=80`
  - `curl http://127.0.0.1:5173/healthz`

验收标准：
- 域名可以打开首页。
- 控制台没有影响使用的红色报错。
- 刷新页面后不会出现“页面尚未完成初始化”提示。

#### 2. 全站 UI 统一

功能作用：
- 让生活收支、订阅、人情往来、设置等页面视觉风格一致。
- 避免按钮、输入框、卡片、弹窗各自为政。

实现方法：
- 统一按钮样式：主按钮、次按钮、危险按钮、幽灵按钮。
- 统一表单样式：输入框、下拉框、日期选择、文件上传。
- 统一卡片样式：边距、圆角、边框、标题层级。
- 统一弹窗样式：宽度、标题、操作区、关闭方式。

验收标准：
- 页面看起来属于同一套设计系统。
- 同类按钮在不同页面表现一致。
- 信息密度合理，不出现大面积无效留白。

#### 3. 基础交互检查

功能作用：
- 确保用户点击、输入、筛选、保存、删除等基础操作都正常。

实现方法：
- 检查所有导航入口。
- 检查所有新增、修改、删除按钮。
- 检查登录后编辑、未登录只浏览逻辑。
- 检查设置面板的数据导入、导出、清空功能。

验收标准：
- 未登录时不能编辑数据。
- 登录后可新增、修改、删除数据。
- 页面切换、弹窗打开关闭、筛选搜索都能正常使用。

#### 4. 部署文档整理

功能作用：
- 让后续本地开发、推送 GitHub、VPS 更新都有固定流程。

实现方法：
- 保留并持续维护：
  - `GITHUB_TO_VPS_FULL_GUIDE.md`
  - `NEWBIE_DEPLOYMENT_GUIDE.md`
  - `UPDATE_WORKFLOW.md`
  - `PRODUCTION_DEPLOYMENT_CHECKLIST.md`

验收标准：
- 新功能完成后可以按文档完成：
  - 本地提交
  - 推送 GitHub
  - VPS 拉取
  - Docker 重建
  - 域名验证

## V0.23 生活收支增强版

### 目标

把生活收支从简单账单列表升级为家庭消费分析系统，重点解决支付宝、微信账单导入，男女双方开销归属，历史消费查看，以及家庭主要支出识别。

### 需要完成的内容

#### 1. 支付宝 / 微信账单导入

功能作用：
- 支持用户导入主要账单来源。
- 减少手动录入成本。

实现方法：
- 导入前选择账单来源：
  - 支付宝
  - 微信
- 根据不同来源解析 CSV / Excel 字段。
- 识别交易时间、交易对象、商品说明、金额、收支类型、支付方式。

操作方式：
- 进入设置面板。
- 选择“导入生活收支”。
- 选择来源：支付宝或微信。
- 选择归属：男方、女方、家庭共同。
- 上传文件。
- 查看导入校验报告。

验收标准：
- 支付宝账单可以正常导入。
- 微信账单可以正常导入。
- 导入后能看到成功、失败、重复数量。

#### 2. 男方 / 女方 / 家庭共同归属

功能作用：
- 区分家庭成员各自开销。
- 分析家庭支出承担结构。

实现方法：
- 账单增加 `owner` 字段。
- 可选值：
  - 男方
  - 女方
  - 家庭共同
- 导入时批量写入归属。
- 手动新增和编辑账单时也可修改归属。

验收标准：
- 生活收支页面可以按归属筛选。
- 统计中可以看到男方、女方、共同支出对比。

#### 3. 历史消费数据查看

功能作用：
- 不只看当前月，可以回看任意月份和年度数据。

实现方法：
- 增加月份选择器。
- 增加年度汇总入口。
- 月份数据和总数据分开展示：
  - 月份数据：当前选择月份的收入、支出、结余。
  - 总数据：全部导入账单的累计收入、累计支出、累计结余。

验收标准：
- 可以查看历史月份。
- 可以查看年度汇总。
- 月份统计不会和总统计混淆。

#### 4. 家庭消费动向分析

功能作用：
- 看清楚钱主要花在哪里。
- 特别关注房贷、孩子、订阅、固定支出。

实现方法：
- 增加分类统计。
- 增加固定支出识别。
- 增加房贷标签。
- 增加孩子相关标签。
- 增加主要支出提示。

验收标准：
- 能看出最大支出分类。
- 能看出房贷占收入比例。
- 能看出孩子相关支出趋势。

#### 5. 生活收支单独清空

功能作用：
- 允许只清空生活收支数据，不影响订阅、人情、笔记等数据。

实现方法：
- 设置面板增加“清空生活收支”按钮。
- 点击后要求二次确认。
- 只删除 `bills` 数据。

验收标准：
- 清空生活收支后，其他模块数据保留。

## V0.24 订阅管理成熟版

### 目标

把订阅从项目列表升级为续费提醒、到期管理、年度成本分析工具。

### 需要完成的内容

#### 1. 订阅项目紧凑展示

功能作用：
- 避免订阅项目多了以后页面过长。

实现方法：
- 订阅卡片采用自适应双栏或多栏布局。
- 压缩无效说明文字。
- 重点展示名称、金额、到期时间、状态、提醒等级。

验收标准：
- 多个订阅项目时不需要大量下拉。
- 临期项目能明显看出来。

#### 2. 到期提醒机制

功能作用：
- 防止忘记续费或忘记取消自动扣费。

实现方法：
- 按时间分组：
  - 已过期
  - 今天到期
  - 7 天内
  - 30 天内
- 临期订阅使用红色警告。
- 支持浏览器提醒。
- 支持邮件提醒。

验收标准：
- 到期订阅显示在明显位置。
- 邮件提醒配置后可正常发送测试邮件。

#### 3. 邮件通知正式化

功能作用：
- 即使不打开网站，也能收到订阅到期提醒。

实现方法：
- 使用 Resend API。
- VPS 配置环境变量：
  - `RESEND_API_KEY`
  - `SUBSCRIPTION_EMAIL_FROM`
- 使用定时任务执行邮件扫描脚本。

验收标准：
- 测试邮件可以发送。
- 到期订阅可以触发邮件提醒。

#### 4. 订阅成本分析

功能作用：
- 了解订阅一年花多少钱。

实现方法：
- 计算月成本。
- 计算年成本。
- 按归属分类：
  - 个人
  - 家庭
  - 工作
  - 孩子
- 按支付渠道分类：
  - 支付宝
  - 微信
  - 银行卡
  - App Store

验收标准：
- 能看到当前订阅总月成本。
- 能看到当前订阅总年成本。
- 能看出高成本订阅。

## V0.25 数据安全与真实登录版

### 目标

把当前本地存储原型升级为可长期使用的真实系统，解决数据安全、账号登录、跨设备访问、备份恢复问题。

### 需要完成的内容

#### 1. 真实登录系统

功能作用：
- 不再使用本地演示账号。
- 用户必须登录后才能编辑数据。

实现方法：
- 增加后端服务。
- 用户账号存入数据库。
- 密码使用哈希存储。
- 使用 Session 或 JWT 保持登录状态。

验收标准：
- 未登录只能浏览。
- 登录后可以编辑。
- 账号密码不再写死在前端代码里。

#### 2. 数据库持久化

功能作用：
- 避免浏览器缓存清空导致数据丢失。
- 支持换电脑、换浏览器继续使用。

实现方法：
- 第一轮先增加受密钥保护的服务器数据 API，用 Docker 数据卷保存整站 JSON 数据，作为从 localStorage 迁移到数据库前的稳定底座。
- 第二轮再引入 SQLite，把当前 JSON 快照拆分为真实数据表。
- 后期可升级 PostgreSQL。
- 数据表包括：
  - 用户
  - 生活收支
  - 订阅
  - 人情往来人物
  - 人情往来记录
  - 事项
  - 笔记
  - 收藏

验收标准：
- 当前第一轮：可以在设置页把浏览器数据保存到服务器，也可以从服务器恢复到浏览器。
- Docker 重建后 `/app/data/personal-hub-data.json` 仍然存在。
- 后续数据库轮次：数据不再依赖浏览器 localStorage。

#### 3. Docker 数据卷

功能作用：
- 防止容器重建时数据库丢失。

实现方法：
- 在 `docker-compose.yml` 中配置 volume。
- 当前第一轮把服务器数据文件放入 `/app/data/personal-hub-data.json`。
- 后续 SQLite 数据库文件也放入同一个持久化目录。

验收标准：
- `docker compose up -d --build --force-recreate` 后数据不丢。

#### 4. 自动备份

功能作用：
- 防止误删、服务器故障、数据库损坏。

实现方法：
- 增加备份脚本。
- 每天或每周导出数据库备份。
- 保留最近若干份备份。

验收标准：
- 可以手动备份。
- 可以从备份恢复。

## 后续执行规则

1. 每次只推进一个版本，不同时混做多个阶段。
2. 每个版本先修稳定性，再加功能。
3. 每次改动完成后必须运行项目检查：
   ```powershell
   node scripts/check.mjs
   ```
4. 每次上线前按部署检查清单执行。
5. 每个版本完成后更新版本号。
6. 每个版本完成后记录主要变化，方便后续维护。

## 下一步

下一轮开发继续推进 `V0.25 数据安全与真实登录版`。

优先任务：

1. 将生活收支、订阅、人情往来等业务数据逐步拆分进 SQLite 表。
2. 增加多设备首次登录时的数据合并引导，避免本地旧数据被服务器数据直接覆盖。
3. 增加用户数据迁移工具，支持把旧全局数据指定迁移给某个用户。
4. 保留现有导入导出能力，作为上线前后的迁移和兜底方案。
### V0.25 本轮补充：用户数据迁移工具

已完成内容：
- 管理员用户管理弹窗增加“迁移旧数据”按钮。
- 后端增加 `/api/auth/users/:id/migrate` 接口。
- 支持把旧全局数据 `/app/data/personal-hub-data.json` 复制到指定用户数据文件 `/app/data/users/user-用户ID.json`。
- 迁移前会要求确认，目标用户已有服务器数据时按确认结果覆盖。
- 用户列表接口会返回旧全局数据状态，方便判断是否需要迁移。

作用：
- 解决从单用户旧数据升级到多用户账号体系后，旧数据不知道归属哪个账号的问题。
- 管理员可以把旧数据指定给主账号或家庭成员账号。
- 迁移后，不同用户继续使用各自独立的数据文件，为后续真正多端同步打基础。
